🛡️ Rollout Guard — What it is & how it fits

Pre-requisiteRequires new_ux feature flag active (set by Staffbase). Without it, new UX only accessible at /web/ path.
What it controlsWho sees new UX on standard paths (without /web/). Options: selected users/groups or all users. Per-feature toggles: Search, My Groups, Bookmarks, Plugins, Directory.
Where it livesStudio → Settings → New UX Rollout. Only visible when new_ux flag is on.
Brand Settings impactNone directly. The settings split (old/new) exists whenever new client is activated. Rollout Guard only controls which users see which UX.
Beta purposePrimarily a Beta-phase tool. May be used post-GA for specific feature rollouts. Long-term may be replaced by feature flags.
Applies to statesStates ③ and ④ only. Not applicable to ①② (no new client) or ⑤⑥ (new customers, always on new UX).

Key Findings & Design Implications

Research across #temp-proj-advanced-styling-concept, #proj-system-ux-trio, #team-strangler-things, and Confluence surfaced six distinct customer states that each have a different combination of brand settings and CSS editors visible in Studio.

⚠️ The "two CSS editor problem" appears twice — in State ② (old customer, old client, using Content Designer) and State ⑤ (new customer, new client, but news/legacy content not yet migrated). Both cases use a tabbed interface, confirmed in the #temp-proj-advanced-styling-concept alignment.

State ② is the hardest to design for right now. Old customers who adopted the Content Designer but don't yet have the new client end up with two CSS editors inside the Old UX settings. The team aligned that these should be tabs, not separate text areas. Tab labels should describe the affected content, not "Old/New UX" (which Frank Erler and Maximilian flagged as meaningless to admins).

States ③ and ④ have a clean separation. Once the new client is activated, old CSS stays in old settings, new CSS goes in new settings — no tabs needed. This was Frank Erler's proposal and was validated by Szymon Soltysiak: "The 2 CSS editor problem only applies to Old Customer, Old UX + Modern pages."

State ⑤ is the underappreciated edge case. New customers still need the old CSS editor for News articles, which are estimated to stay unrenovated until at least H1 2027. Without it, News layout customization becomes impossible for new customers — Mikael Head called this "a no-go."

The Rollout Guard doesn't change the brand settings configuration — it only controls who sees which UX on standard paths. The settings split is determined purely by whether the new_ux flag is active. The Rollout Guard applies to States ③ and ④ only.

The long-term direction is State ⑥: one settings section, one CSS editor, guided by the 3-Tier customization model (Tier 1: Studio Brand Settings → Tier 2: Design tokens → Tier 3: Contracted CSS editor). Target is 70% of customers completing migration without needing Tier 3 CSS at Open Beta.
